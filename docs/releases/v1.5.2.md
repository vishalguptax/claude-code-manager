# v1.5.2

_Released 2026-04-22_

Repairs the multi-account switcher: identity now survives Claude CLI's background token rotation, switching preserves your project history + migration flags instead of rolling them back, and the two-file swap is crash-safe with rollback.

## 🐛 Bug Fixes

- account: repair profile switching — identity match, merge-preserve, atomic swap ([914f8b9](https://github.com/vishalguptax/claude-code-manager/commit/914f8b9a28ae8feb684c7d14e82027a1a7f3aae0))

### Highlights

Eleven compounding bugs in the multi-account switcher, all rooted in three wrong assumptions. This release replaces them with three correct ones.

**Active-profile detection follows identity, not bytes**
`getActiveProfileSlug` used only the SHA-256 of `~/.claude/.credentials.json` — so any background token refresh by Claude CLI made the currently-active profile look unsaved. That caused a cascade:

- The Account tab re-offered "Save profile" for an already-saved account
- Clicking it created `agent-prinics-2`, `agent-prinics-3`, … forever
- The switcher's self-switch short-circuit (`pick.slug === activeSlug`) mis-fired because `activeSlug` was `null`, so clicking your own row ran a full destructive swap on your live state

Detection now cascades through three matchers: byte-identical hash → `userID` (stable across token rotations) → email (fallback for older snapshots). Snapshots store `userID` in `profile.json` going forward. The cascade resolves the correct slug even after the CLI has refreshed tokens a dozen times.

**Switch merges identity into live state**
`switchProfile` used to overwrite `~/.claude.json` wholesale from the snapshot. `~/.claude.json` holds ~45 top-level keys — only two of them (`oauthAccount`, `userID`) are identity; the rest are projects, `numStartups`, migration flags (`sonnet1m45MigrationComplete`, `opusProMigrationComplete`, `migrationVersion`, `hasCompletedOnboarding`), MCP config, onboarding flags, and dozens of caches. Switching rolled all of that back to whatever was captured at save time. The migration flags in particular made Claude CLI re-run old migrations against fresh state, which users experienced as "my Claude settings got broken."

Switching now reads the live file, swaps **only** `oauthAccount` + `userID` from the snapshot, and writes the merged object back. `~/.claude/.credentials.json` is still swapped wholesale — it contains only OAuth tokens, so a clean replacement is correct.

**Two-file swap is now crash-safe**
The old swap did two sequential `renameSync` calls with no rollback, so a failure between them left `~/.claude.json` pointing at one account and `~/.claude/.credentials.json` still holding another account's tokens. The new swap copies both live files to `.bak` siblings before the first rename, and the second rename's error handler restores from backup so the pair can't end up desynced.

**Saves + updates are race-safe**
`saveProfile` and `updateProfile` now hash `.credentials.json` before and after the paired read, retrying once when the hash moves (Claude CLI mid-refresh). Snapshots that mix identity from one token generation with credentials from another are no longer possible.

**Dedupe on save**
Saving a profile when one already exists for the current `userID` returns `already-saved` with the existing slug. The Account tab and switcher now offer to refresh (Update) the existing slot instead of silently minting a duplicate. This is the direct fix for the profile-accretion pattern that the old hash-only detection produced.

**Stale-token warning in the confirm dialog**
Switching to a snapshot whose access token has already expired prepends a ⚠-prefixed notice to the confirm modal, with a human "N days ago" measurement. Users get a signal that a `/login` may be required after the swap (refresh tokens rotate on use, so long-stale snapshots may have no valid refresh path left).
